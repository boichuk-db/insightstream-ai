# Documentation Actualization — Design

> Roadmap item: `docs/architecture/PLAN.md` §🔥 Implement Soon #13 / at-a-glance table #19.

## Problem

`README.md` and the project's docs haven't kept pace with recent work (Team-as-Tenant, Stripe billing, the AWS migration, the BullMQ worker split, the P0–P1 UI passes). `README.md` currently describes a stack that no longer exists (Railway + Supabase, Vanilla-TS widget, no Team/Stripe/landing/e2e mention). There's no single readable, up-to-date entry point to the project — just `PLAN.md`, 34 scattered spec docs and 32 plan docs under `docs/superpowers/`, and an 8-page `system-architecture.drawio` of unconfirmed accuracy.

## Goal

A compromise between "just fix README + drawio" and "build a full public product doc site": a real but lightweight documentation site, primarily for the solo maintainer's own onboarding/recall, that is also comfortably readable from a phone. Not aimed at external customers/investors.

## Approach

**Tool:** Docusaurus, deployed to GitHub Pages. The repo (`boichuk-db/insightstream-ai`) is already public, so a public GitHub Pages site introduces no new exposure — it mirrors `PLAN.md`, which is already public in-repo. Rejected alternatives: Mintlify/GitBook (ties to a third-party host + free-tier limits for a solo internal doc site is disproportionate); "just markdown in `docs/` read via GitHub" (works, but no real navigation/search, and the user specifically wants a comfortable phone-browsing experience, which GitHub's raw markdown view doesn't give).

**Location & build:** `apps/docs`, a normal member of the pnpm workspace — participates in `pnpm dev/build/lint/typecheck` like every other app. A new GitHub Actions workflow (`.github/workflows/docs.yml`) builds and publishes to GitHub Pages on every push to `main` that touches `apps/docs/**` or `docs/architecture/**` (path-filtered, so unrelated commits don't trigger a deploy).

## Site structure

| Section | Content | Source |
|---|---|---|
| **Overview** | What the product is, stack table, repo layout | Fresh write, cross-checked against current code (not the stale README) |
| **Architecture** | The 8 `system-architecture.drawio` pages, each as an embedded SVG + short text explaining what it shows | Exported from `.drawio` (see Diagrams pipeline) |
| **Modules** | One page per `apps/*` (api, web, widget, landing, e2e) and `packages/*` (database, shared-types, config): purpose, key internal modules/domains, where to look in code | Fresh write, read from current code |
| **Ops** | Deploy process (`deploy-api.sh`, Amplify/Vercel parallel run), env vars by name (no values), runbooks | `DEPLOYMENT.md` + `PLAN.md`, reconciled with current reality |
| **Roadmap** | Short current-state summary (what's in 🔥, what's in 🟡) + a direct link to `docs/architecture/PLAN.md` on GitHub | Link only — **no table copy**, per `PLAN.md`'s own "don't duplicate across places" rule |

## Diagrams pipeline

New script `scripts/export-diagrams.sh` (drawio CLI, headless) exports all 8 pages of `system-architecture.drawio` to SVG under `docs/architecture/diagrams/*.svg`, committed alongside the `.drawio` source. Docusaurus Architecture pages embed these SVGs directly (`<img>`). The `.drawio` file stays the source of truth; SVGs are a generated, committed artifact regenerated by the script whenever the diagram changes.

**Update rule extension:** `PLAN.md`'s existing "Update rule" (any architecture change updates `PLAN.md` + fixes stale diagrams in the same PR) gains one step: after editing `.drawio`, run `scripts/export-diagrams.sh` and commit the regenerated SVGs in the same PR. This line gets added to `PLAN.md`'s header note and to `CLAUDE.md`'s Architecture Documentation section.

## README.md

Cut down to a repo entry point only: one-line description, quickstart (clone → install → `docker compose up` → `pnpm dev`), the dev-ports table, and a link to the Docusaurus site for everything else. Architecture / Features / Deployment sections currently in README move to the site and are deleted from README — one source of truth, not two to keep in sync.

## Out of scope (deliberate)

- Auto-generated API reference (Swagger/OpenAPI) — not requested, adds real setup cost.
- ADR journal — not requested.
- Migrating the 34 spec docs / 32 plan docs under `docs/superpowers/` into the site — they stay as the historical record they already are; the site links out to `PLAN.md`, not into that archive.
- Custom domain for GitHub Pages — uses the default `*.github.io` URL.

## Effort

Not previously estimated in `PLAN.md` (flagged as needing this brainstorming pass first). Expect: Docusaurus scaffold + CI workflow (~half day), diagram export script + 8 SVGs (~1–2 hours), Overview/Ops/Roadmap content (~1–2 hours, mostly reconciling stale facts), 8 Modules pages (~half day, one read-through per app), README rewrite (~30 min). Roughly **1.5–2 days** total — to be refined in the implementation plan.
