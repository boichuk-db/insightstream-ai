## Project Constraints

The filter every roadmap item must pass:

1. **Infrastructure cost as close to zero as possible** (free tier where feasible).
2. **Learning modern backend/cloud architecture hands-on is a first-class goal.** EC2 (not Fargate), BullMQ, Socket.io, and the AWS migration itself are **deliberate learning choices** ŌĆö not oversights to optimize away. Each item below that names one of these carries its graduation trigger instead of a "replace with managed service" recommendation.
3. **No enterprise complexity before it pays for itself.**
4. **An item enters ­¤öź Implement Soon only if it fixes a real, current problem.** Everything else goes to ­¤¤Ī Future with an explicit adoption trigger.

### 2. ~~Fix the OAuth personal-team gap~~ ŌĆö Ō£ö Done (2026-07-03)
**Original premise was stale:** `oauthLogin()` has created a personal team for new Google/GitHub users since the OAuth feature first landed (commit `4169358`) ŌĆö there was never a gap for new OAuth signups. The idempotent `TeamsService.ensurePersonalTeam()` (which also migrates orphan `teamId IS NULL` projects) already served as a lazy backfill via `GET /teams`.
**What was done:** hardened rather than re-implemented ŌĆö both signup paths (`register` + `oauthLogin`) now call the single idempotent `ensurePersonalTeam()` hook instead of the raw `createPersonalTeam()`, removing the duplication and the double-team risk; `createPersonalTeam()` is now `private` so the safe hook is the only entry point. No proactive DB migration written: the lazy heal already covers stragglers and there are zero production users.
**Deliberately skipped:** wrapping `user.create` + team creation in a cross-service DB transaction ŌĆö the lazy `ensurePersonalTeam` heal makes the transient teamless state self-recovering, so a transaction is disproportionate plumbing at this stage (constraint #3). Revisit if signups ever run under real concurrency.
**Type:** defect fix (mostly pre-existing).

### 3. ~~Stripe webhook idempotency + ordering~~ ŌĆö Ō£ö Done (2026-07-03)
Dedup via a new `StripeEvent` log (Stripe event id as PK ŌåÆ recording an event is an idempotent insert; retries/replays are inert). Ordering via a new `users.lastStripeEventAt` stamp (nullable, epoch DB default): every subscription mutation is a single **atomic conditional** `UPDATE ŌĆ” WHERE "lastStripeEventAt" IS NULL OR "lastStripeEventAt" <= :eventCreated`, so an out-of-order or delayed event (`updated` after `deleted`) can no longer resurrect a canceled plan ŌĆö and the check-and-set being one statement closes the race under concurrent delivery. (Column left **nullable** deliberately: a NOT NULL variant makes dev `synchronize` fail with `SET NOT NULL` on the already-populated `users` table; the `IS NULL` arm keeps the guard correct for any un-stamped row.) Dispatch centralized in `StripeWebhookService.handleEvent(event)`; the controller just verifies the signature and delegates. Migration `1774840000000-AddStripeEventsAndOrdering`. Seeds the future subscription-history table.

### 4. ~~Self-healing AI sweep~~ ŌĆö Ō£ö Done (2026-07-03)
**Problem:** pending AI jobs live only in Redis (a container on the same EC2 instance). Instance loss or a crash means analyses silently never happen.
**Done:** new `AiSweepService` (`apps/api/src/modules/ai/ai-sweep.service.ts`), a `@Cron('*/5 * * * *')` sweep that re-enqueues `sentimentScore IS NULL` feedback created in the **(15 min, 24 h)** window via the existing `AiQueueService` (job data reconstructed from the row + owner plan, cached per run; `aiLevel: 'none'` skipped). Idempotent by nature ŌĆö the 15-min age threshold guarantees no live job exists, so **no jobId dedup**; makes queue durability, DLQ, and crash recovery largely moot. Rows still `NULL` past 24 h are logged as **abandoned** (bounded poison-message cost, no schema change). Design + plan: `docs/superpowers/specs/2026-07-03-self-healing-ai-sweep-design.md`. **Deferred (design follow-ups):** a partial index `feedbacks (createdAt) WHERE sentimentScore IS NULL` (YAGNI at current scale) and, when the worker splits (#5), the `ai-sweep` cron must not boot in `WORKER_MODE`.
**Type:** resilience.

### 5. ~~Separate BullMQ worker process~~ ŌĆö Ō£ö Done (2026-07-09)
**Problem:** the AI worker (concurrency 3) shared one Node process and the t3.micro's burstable CPU credits with HTTP + WebSocket ŌĆö sustained AI load could silently throttle the API.
**What was done:** `apps/api/src/main.ts` now branches on `WORKER_MODE=1` ŌĆö worker mode calls `NestFactory.createApplicationContext(WorkerModule)` (no HTTP, no Socket.io, no `ScheduleModule`) instead of the full HTTP bootstrap. New `WorkerModule` (`apps/api/src/worker.module.ts`) imports only `TypeOrmModule`/`BullModule` (via new shared `getTypeOrmConfig`/`getBullConfig` factories in `apps/api/src/config/` ŌĆö the worker passes `migrationsRun: false` so both containers never race migrations on deploy), a new worker-only `AiWorkerModule`, and a new `WorkerEventsModule`. `AiSweepService` (the cron sweep) was moved out of `AiModule` into its own `AiSweepModule`, imported only by `AppModule` ŌĆö the worker process cannot boot it even by accident. Since a worker process never has a live Socket.io server, `AiProcessor` no longer depends on the concrete `EventsService`; it depends on a new `FeedbackEventsPublisher` DI token instead. The HTTP process binds the token to the existing `EventsService` (unchanged behavior); the worker process binds it to a new `RedisFeedbackEventsPublisher`, which emits through `@socket.io/redis-emitter` into the same Redis channel `@socket.io/redis-adapter` already uses ŌĆö same room (`team-{id}`) and event name (`feedbackUpdated`), so the dashboard client sees no difference. Deployed as a second `docker run` (`insightstream-worker`, no `-p`, no HTTP-only env vars) alongside the existing `insightstream-api` container in `scripts/docker-run.sh`. Design: `docs/superpowers/specs/2026-07-09-separate-bullmq-worker-design.md`. Plan: `docs/superpowers/plans/2026-07-09-separate-bullmq-worker.md`.
**Bug caught by real end-to-end testing, not typecheck/unit tests:** the initial implementation (through the plan's Task 10) still left `AiProcessor` reachable from `AppModule` ŌĆö `AiModule` (which provided `AiProcessor` alongside `AiService`/`AiQueueService`) was still imported transitively via `FeedbackModule` and `AiSweepModule`, so NestJS kept instantiating a full second `AiProcessor` BullMQ consumer inside the HTTP process, defeating the entire point of the split (both processes were racing the same queue; the HTTP process's CPU was still doing AI analysis). Caught only by actually running both processes against real Postgres/Redis, submitting real feedback, and checking which process's logs showed "Processing AI analysis" ŌĆö unit tests and typecheck were green throughout because nothing in them exercised cross-module reachability. Fixed by extracting `AiProcessor` into a new `AiWorkerModule` (`apps/api/src/modules/ai/ai-worker.module.ts`), imported only by `WorkerModule`; `AiModule` now only provides `AiService`/`AiQueueService` for enqueue-only consumers. Re-verified end-to-end after the fix: HTTP process log has zero `AiProcessor` references, worker process log shows the job processed, and the WS client still receives `feedbackUpdated` live via the Redis-emitter relay.
**Known gaps, deliberately deferred (surfaced during code review, not blockers):**
- No graceful shutdown wiring (`app.enableShutdownHooks()`) anywhere in `apps/api` ŌĆö pre-existing, but now more consequential: `docker stop` on the worker container (e.g. every redeploy) sends `SIGTERM` then `SIGKILL` with no drain period, so an in-flight `AiProcessor` job gets hard-killed mid-analysis instead of completing or cleanly re-queuing. Revisit if job loss during redeploys becomes a real, observed problem (the existing self-healing AI sweep, Ō£ö #4, will eventually re-enqueue an abandoned job anyway, bounding the damage to a delayed analysis, not a lost one).
- Neither the `insightstream-api` nor the new `insightstream-worker` container receives `SENTRY_DSN` via `docker-run.sh`'s `-e` flags (pre-existing gap for the API container, not introduced here) ŌĆö worker-process crash loops would currently be invisible to Sentry, visible only via `docker logs`/restart counts.
**Type:** performance isolation.

### 6. ~~Redis cache for the JWT user lookup~~ ŌĆö Ō£ö Done (2026-07-06)
**Problem:** `JwtStrategy.validate()` read Postgres on every authenticated request ŌĆö the hottest query in the system, negating the stateless-JWT scaling rationale.
**What was done:** new `@Global()` `RedisModule`/`RedisService` (`apps/api/src/redis/`) ŌĆö a thin `get`/`set`/`del` wrapper over `ioredis` that catches client errors and treats them as a miss/no-op (fail-open), logged once via `Logger.warn`. The `ioredis` client is constructed with `{ maxRetriesPerRequest: 1, enableOfflineQueue: false, commandTimeout: 500 }` so a Redis outage fails fast (bounded ~500ms) instead of ioredis's default retry/backoff behavior (which would otherwise add up to ~10s of latency per call before falling through) ŌĆö this was caught and fixed during code review, since it directly undermined the point of protecting the hot path. `JwtStrategy.validate()` is now read-through: cache hit returns `{id, email, role}` from Redis without touching the DB; miss falls back to `UsersService.findOneById`, then warms the cache. TTL 30s, no active invalidation (accepted bounded revocation latency ŌĆö a banned/role-changed user keeps old access for up to 30s). `JwtStrategy` additionally wraps its own Redis read in a local try/catch and validates the parsed cached value's shape (`id`/`email`/`role` all strings) and identity (`id` must match the verified JWT `sub` claim) before trusting it ŌĆö defense-in-depth so a future regression in `RedisService`'s own fail-open guarantee, or a malformed/wrong-user value ending up under the cache key, can't turn into either a hard auth failure or a trusted-but-wrong authenticated principal. First app-level cache use of Redis in this codebase; the connection is shared via the new `RedisService` rather than hand-rolling a 4th independent client (Socket.io adapter, BullMQ, and throttler storage each already have their own). Design: `docs/superpowers/specs/2026-07-06-jwt-user-lookup-redis-cache-design.md`.
**Deliberately out of scope:** active cache invalidation on user save/role-change/delete; consolidating the other three existing Redis clients onto `RedisService`.
**Type:** performance.

### 8. ~~Widget: versioned URL now, weight later~~ ŌĆö Ō£ö Done (2026-07-11)
**Problem:** `widget.iife.js` was a 380 KB React bundle served from an unversioned S3 URL ŌĆö any breaking change instantly broke every customer site with no rollback; the weight hurt customers' page scores (the widget *is* the product).
**URL versioning ŌĆö done (2026-07-09):** S3 key moved from the mutable `widget.js` to the mutable `v1/widget.js` (major-version-only: a future breaking change ships under a new key, e.g. `v2/widget.js`, while `v1` stays frozen for old integrations). New `scripts/deploy-widget.sh` (build ŌåÆ upload ŌåÆ verify, mirrors `deploy-api.sh`'s conventions) replaces the previously script-less manual `aws s3 cp` command that only lived inside a historical migration plan doc. Rollback is git-based (check out an old commit, re-run the script) ŌĆö no immutable per-build S3 history, a deliberate simplicity trade-off. `./scripts/deploy-widget.sh` has been run for real (`v1/widget.js` live, verified 200 + correct content-type) and `NEXT_PUBLIC_WIDGET_URL` updated + redeployed on both Amplify and Vercel ŌĆö the new URL is live in production, not just repo-side. Design: `docs/superpowers/specs/2026-07-09-versioned-widget-url-design.md`.
**Weight ŌĆö done (2026-07-11):** rewrote `apps/widget` on **Preact** (`preact` + `preact/hooks`, no `preact/compat`), dropping `react`/`react-dom`, `framer-motion`, `lucide-react`, `axios`, and the unused `clsx`/`tailwind-merge`. Bare Preact was chosen over vanilla TS after checking real numbers: plain `react`+`react-dom` alone (before even framer-motion/lucide/axios) is ~46 KB gzip, already over budget, so React in any form was out; Preact's core+hooks add only a few KB and let the JSX structure carry over almost unchanged (low rewrite risk for a single-component, no-list-diffing tree), so hand-rolled vanilla DOM manipulation wasn't worth its extra risk once Preact already cleared the budget by a wide margin. `preact/compat` was not needed ŌĆö the component only used `useState`, so no React-compat shims had to be pulled in. Framer-motion's two animations (panel enter/exit, success-icon bounce) were replaced with plain CSS transitions/keyframes (`animate-success-pop`, `animate-fade-in` in `index.css`); the panel's `AnimatePresence` unmount-on-exit was reimplemented as a small `panelMounted`/`panelShown` state pair with a timeout matching the CSS transition duration. Lucide icons were replaced with 5 hand-copied inline SVGs (`apps/widget/src/icons.tsx`, path data copied from the installed `lucide-react`, ISC-licensed) rather than pulling in `preact/compat` + `lucide-react`'s React runtime. `axios` was replaced with `fetch`. Tailwind CSS (already JIT, already Shadow-DOM-injected via `?inline`) was kept as-is ŌĆö it's not a JS-bundle cost driver at this scale and reduced rewrite risk by keeping the original className strings.
**Result:** `dist/widget.iife.js` **42.4 KB raw / 12.9 KB gzip** (vite's report; `gzip -c | wc -c` gives 12,709 bytes) ŌĆö down from 380 KB / 122 KB gzip, **58% under** the 30 KB gzip budget. `pnpm --filter widget build/lint` and `tsc -b` clean. Verified with a real headless-Chromium script driving the actual built bundle (not just typecheck): Shadow DOM isolation intact (no host-page style leakage, styles confined to the shadow root), trigger ŌåÆ open ŌåÆ autofocus ŌåÆ fill ŌåÆ submit ŌåÆ success, the POST body shape (`{apiKey, content, source: "Widget"}`) unchanged, the 3s auto-close-after-success timer, and the fetch-failure error path all behave identically to the original. One real bug caught by that run and fixed before merge: the autofocus effect initially keyed off `isOpen` and fired before the textarea existed in the DOM (it only mounts a render later, once `panelMounted` flips) ŌĆö fixed by keying the effect off `panelMounted` instead.

## Ō£ģ Keep As-Is

Deliberate decisions and why they stay.

### Architecture & Data
- **Modular NestJS monolith** ŌĆö one developer; module seams give all the evolution room needed. Microservices at this size would be malpractice.
- **PostgreSQL + TypeORM + real FKs**, shared entities via `@insightstream/database` ŌĆö relational fits the domain; FK integrity is doing real work daily.
- **BullMQ** ­¤Äō ŌĆö teaches queue semantics hands-on. The *seam* matters, not the tech; revisit (ŌåÆ SQS) only if consolidating fully on AWS after cutover.
- **Socket.io + Redis adapter** ŌĆö the hard part is already done; scales horizontally today. No reason to touch it.
- **`PLAN_CONFIGS` + `PlanLimitsService`** ŌĆö single source of plan truth. Caveat to remember: `JSON.stringify(Infinity)` ŌåÆ `null`; audit any endpoint that serializes plan configs to the frontend.
- **Current AI flow** (queue ŌåÆ Gemini ŌåÆ write-back ŌåÆ WS emit) ŌĆö the shape is right; the worker split (­¤öź #5) changes *where* it runs, not the flow.
- **Current module boundaries** ŌĆö correct; the tenant fix (Ō£ö #7) is done. Split identity-auth from widget-key-auth only when auth is next touched anyway.

### Platform & Practices
- **EC2 + manual Docker deploy** ­¤Äō ŌĆö the intentional learning path (VPC, SGs, ALB, SSM the hard way). Graduation trigger: the Fargate item in ­¤¤Ī.
- **The AWS migration overall** ­¤Äō ŌĆö the learning goal itself. One discipline: two production stacks must not coexist indefinitely ŌĆö set a cutover-or-park decision date.
- **JWT bearer auth** ŌĆö fine until launch hardening (then refresh tokens + cookies from ­¤¤Ī). The lookup cache (­¤öź #6) removes its main runtime cost.
- **Hand-rolled CORS middleware** ŌĆö explicit, readable, and the public-vs-dashboard branch logic is documented in code. No need for `enableCors()`.
- **Single region eu-north-1** ŌĆö EU data residency is a GDPR feature, not a limitation.
- **Monorepo + Turbo + CI** (lint / typecheck / test / e2e) ŌĆö the strongest part of the developer experience. Do not touch.
- **Honest diagram culture** ŌĆö STUB labels, soft-FK notation, verification dates in `system-architecture.drawio`. Keep it; the update rule at the top of this file exists to protect it.

## Ōøö Retired Recommendations

From earlier reviews ŌĆö kept for history, with reasons.

- **EventEmitter2 domain-event bus** ŌĆö same process, same failure domain: decoupling theater. The durable seam is the queue.
- **Generic cache-aside layer** ŌĆö invalidation bugs pre-PMF; two targeted caches deliver ~90% of the value.
- **`NotificationDispatcher` now** ŌĆö email is the only channel; an abstraction with one implementation is dead weight.
- **`PaymentProvider` abstraction** ŌĆö one provider, and webhooks couple deeply anyway; Stripe lock-in is acceptable.
- **Soft delete everywhere** ŌĆö GDPR favors hard deletes for user data; revisit per-entity only if an undo feature is requested.
- **ASG + Multi-AZ now** ŌĆö no production traffic on AWS yet; moved to ­¤¤Ī behind a revenue trigger.
- **Separate `AIAnalysis` entity** ŌĆö columns on `Feedback` are fine until re-analysis / versioning becomes a feature.
- **`LocalStrategy` for password login** ŌĆö cosmetic Passport uniformity; the inline bcrypt path works.
- **Standalone Dead Letter Queue** ŌĆö subsumed by the self-healing sweep (­¤öź #4), which recovers all loss modes, not just exhausted retries.
- **Numeric architecture ratings (X/10 scores)** ŌĆö theater on a pre-revenue solo project; dropped from this plan, replaced by "does it solve a real problem" as the only test.
